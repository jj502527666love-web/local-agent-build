## 0. 商城标识约定（mall_key，贯穿三端唯一事实源）

| mall_key | 真实平台名（云控端后台可见） | 终端用户可见名 | 域名样例 |
|---|---|---|---|
| `ewei` | eweishop | 云控端自定义（缺省"商城"） | （各连接器自填） |
| `dianda` | 点大商城（iappwx） | 云控端自定义（缺省"商城"） | dip.iappwx.com |

- `mall_key` 是三端（agent-build / agent-admin / agent-desktop）统一约定的字符串枚举，**任一端不一致权限即失效**。

## 1. 点大（iappwx）协议逆向结果

测试环境：`https://dip.iappwx.com`，账号 `D123456` / `D123456`（已实测登录并抓全链路）。
技术栈：ThinkPHP（`?s=/Controller/action` 路由）+ LayUI 后台 + **PHPSESSID Cookie 会话**。与 ewei「自生成 session-id 头 + 无 cookie + AES 密码」完全不同，**协议层无法复用，需新写点大适配器**。

### 1.1 接口契约（已实测）

| 能力 | 点大端点 | 方法 | 入参 | 出参 |
|---|---|---|---|---|
| 登录 | `/?s=/Login/index` | POST 表单 | `username`,`password`(**明文**),`captcha`,`remember` | JSON `{status,msg,url}`；`status==2`=验证码错需刷新 |
| 验证码图 | `/?s=/captcha` | GET | （绑定 PHPSESSID） | PNG 图（**每次登录必填**，高噪点人机型） |
| 会话维持 | — | — | **PHPSESSID Cookie**；`remember=1` 可自动登录延长会话 | — |
| 商品列表 | `/?s=/ShopProduct/index&page=N&limit=M` | GET | `page/limit/name(标题模糊)/status(1上架·0下架)/cid(分类)`（已实测） | layui `{code:0,msg,count,data:[goods]}` |
| 商品编辑页 | `/?s=/ShopProduct/edit/id/{id}` | GET | 路径式 id | 渲染编辑表单（含 `info[*]` 字段） |
| 商品保存 | `/?s=/ShopProduct/save` | POST 表单 | **全量** `id`+`info[*]`+`option[N][*]`+`specs`+佣金字段 | `{status:1,msg:"操作成功",url}`；**缺字段静默清空**（见 §1.3.6） |
| **图片上传** | `/?s=/upload/index` | POST multipart | 字段名 **`file`** | `{status:1,state:"SUCCESS",url:"<绝对URL>",info:{width,height,...}}` |
| 商品分类 | `/?s=/ShopProduct/getcategory` | — | 待补测 | — |
| 取单品 | `/?s=/ShopProduct/getproduct` | — | 待补测（可替代 edit 取详情） | — |

### 1.2 商品 / 图片字段模型（与 ewei 对照）

| 概念 | ewei | 点大 |
|---|---|---|
| 商品标题 | `goods.title` | `info[name]` / 列表 `name` |
| 主图 | `goods.thumb`（**相对路径**，回写后需 COS 前缀反推预览） | `info[pic]`（**绝对 URL**，列表 `pic`） |
| 图集 | `goods.thumbs[]`（相对路径数组） | `info[pics]`（**逗号分隔的绝对 URL 串**） |
| 详情图 | `goods.content`（富文本 HTML，内嵌 `<img>`） | `info[detail]`（**块状 JSON**：`[{id,temp:"richtext",params,content}]`，非 HTML） |
| 多规格/SKU | `options[]`+`specs[]`（回写须带全量 id 防物理删除） | `option[N][*]`（含 `option[N][pic]`=SKU 图）+ `specs` + `guigedata`（列表 `guigedata`） |
| 上传返回 | `{path(相对),url(绝对),width,height,id}` | `{url(绝对),info:{width,height,extension,hash,bsize}}` |
| 回写基底 | 取详情完整 goods 为基底（缺字段会整单失败/误删 SKU） | 同样需"取全量 → 改图字段 → 整体提交"（`info[*]`+`option[N][*]`+佣金字段很多） |

### 1.3 关键差异（影响适配器设计）

1. **登录需验证码（每次）**：点大不能像 ewei 用存的密码静默重登。→ 适配器登录必须支持"拉验证码图 → 用户输入 → 提交"的交互式流程（见 §4.4）。
2. **会话靠 Cookie**：现有 `ewei-client` 用 `fetch` 不带 cookie，点大适配器需引入 **cookie jar**（持久化 PHPSESSID/remember，减少验证码频次）。
3. **无需切门店**：点大登录后 `?s=/ShopProduct/index` 直接列出该商户(aid)全部商品，**不像 ewei 必须先 `shops/switch` 选店**。点大有"门店(mendian)"概念（`info[bind_mendian_ids][]`），但只是商品绑定维度，不是列商品的前置上下文。→ 适配器的 `listShops/switchShop` 对点大可为空实现/隐藏（见 §4.2 能力位）。
4. **图片是绝对 URL**：上传直接返回绝对 URL，塞进 `info[pic]` 即可，无 ewei 的相对/COS 反推。
5. **详情是块状 JSON**：替换/追加详情图要构造 `temp:"richtext"` 的图片块，而非拼 `<img>` HTML。
6. **保存是「全量替换」，缺字段静默清空（已实测·最危险的坑）**：`POST /?s=/ShopProduct/save` 成功返回 `{status:1,msg:"操作成功",url}`。**实测：只传 `id+info[pic]` 的最小提交也返回"操作成功"，但会把 name/pics/库存/规格 等所有未提交字段清空**（比 ewei 更坑——还假报成功）。→ dianda-adapter **必须先取该商品完整数据 → 只改目标图片字段 → 整体回提全部字段（实测该商品约 267 个命名字段）**。安全基底：编辑页 `/?s=/ShopProduct/edit/id/{id}` 服务端渲染已带全部字段当前值，且内嵌 `var product` 完整商品对象（@HTML 内）；适配器据此重建 save 表单（`info[*]`+`option[N][pic]`+`specs`+`info[detail]`块状+`commissiondata*[ID][*]`），等价并强于 ewei 的 `buildEditPayload`。**实测已跑通端到端：上传(返回绝对URL)→设 `info[pic]`→全量保存→列表核对生效→还原，全部成功。**
7. **上传走点大后台启用的存储**：`/?s=/upload/index` 是点大后台 webuploader 自己用的统一附件接口，遵循后台「附件/存储设置」的驱动（本地/OSS/COS/七牛/又拍…）落盘并返回对应 URL。实测该测试号返回本地磁盘 URL（`dip.iappwx.com/upload/5/...`）=当前用本地存储；商户切云存储则同接口返回云存储域名 URL。含义：我们不自管存储，回写的就是商户存储里可公开访问的绝对 URL；同样受点大后台**文件大小/类型限制**约束（适配器需"过大压缩 JPEG 重试"兜底，同 ewei）。

> 实现阶段第一步：在编辑页改一张图后点保存，抓 `?s=/ShopProduct/save` 的完整 POST body，定位最小必填集与多规格防误删点（本次逆向未做破坏性保存）。

## 2. 总体设计：商城注册表（单一事实源）

三端各维护一份注册表，把"某商城的所有平台相关常量"收敛到一处：

- **agent-desktop（主进程）**：`mall_key → MallAdapter`（协议适配器）。
- **agent-build（一级授权）**：`mall_key` 枚举（用于授权关联表的合法值校验）。
- **agent-admin（二级授权+命名）**：`mall_key → { realName, settingKey: '{mall}_shop_mall_name', policyKey: 'allow_{mall}_shop' }`。

新增第 N 商城：三端注册表各加一行 + 桌面端加一个 `XxxAdapter` 实现 + （点大类）抓协议。**不改任何表结构、不改下发协议骨架。**

## 3. 授权链路泛化（目标B：按商城独立授权）

### 3.1 一级授权（agent-build 授权管理端）— 改为关联表

现状：`authorized_clients.can_use_ewei_shop` 单布尔（migration `2026_06_20_000001`）。
**定案：新建关联表，不再加布尔**（因为"以后还会加更多商城"）。

- 新 migration（遵守铁律：只新增、不改旧、只用 `Schema::`/`DB::`）：
  ```
  client_mall_authorizations(id, client_id, mall_key VARCHAR, authorized BOOLEAN DEFAULT 0,
                             created_at, updated_at)  唯一索引(client_id, mall_key)
  ```
  同一 migration 内回填：`can_use_ewei_shop=1` 的客户端 → `(client_id,'ewei',1)`。**保留旧列 `can_use_ewei_shop` 只读不动**。
- `BuildRequestController` auth-check 返回结构：
  - **保留旧字段** `can_use_ewei_shop`（= mall 授权里的 ewei 值）给未升级的云控端。
  - **新增** `mall_authorizations: { ewei: bool, dianda: bool }`（按授权关联表聚合）。
- `BuildClientController`：`setEweiShop/batchSetEweiShop` 泛化为 `setMallAuth/batchSetMallAuth`（入参加 `mall_key` 枚举校验 `in:ewei,dianda` + `authorized`），upsert 关联表；**保留旧 `set-ewei-shop` 路由**映射到 `mall_key='ewei'` 兼容。`index()` 的客户端列表 join 出每个 client 的 mall 授权 map。
- 前端 `Clients.tsx` + `api/clients.ts` + `types`：单 `can_use_ewei_shop` 列改为**按商城渲染多列/多开关**（ewei 列、点大列各一个 Switch；批量按钮带商城选择）。

### 3.2 二级授权（agent-admin 云控端）— per-user 按商城

现状：`EweiShopAuthorization`(单值一级判定) + `QuotaService` 单 `allow_ewei_shop` policy + `ClientController` 下发单 `allow_ewei_shop`。

- `Services/CloudBuild/EweiShopAuthorization.php` → 泛化为按 `mall_key` 的 map：**一次探测 auth-check 拿回全部商城授权位** `{ewei,dianda}` 整体缓存（FRESH/PENDING/LKG），新增 `isAuthorized(mall_key)`。`forget()` 清全部商城键。
- `QuotaService.php` defaults：增 `allow_dianda_shop => false`（保留 `allow_ewei_shop`）。policy 合并机制按 `policy_key` 自由扩展，无需改逻辑；per-user 授权 = `PermissionPolicy(policy_key='allow_{mall}_shop')`。
- `ClientController`（`myPermissions` + `myQuotas` 两处重复逻辑抽成一个私有方法）下发：
  - **新增结构** `shops: { ewei: {allowed, mall_name}, dianda: {allowed, mall_name} }`，其中 `allowed = 二级 policy AND 一级 isAuthorized(mall)`。
  - **保留旧平铺字段** `allow_ewei_shop` + `ewei_shop_mall_name`（= shops.ewei 的值）给旧桌面端兼容。

## 4. 桌面端改造（agent-desktop）

### 4.1 连接器加 `platform` 维度（本地 sqlite 迁移）

- `ewei_connectors` 表加列 `platform TEXT NOT NULL DEFAULT 'ewei'`：
  - `main/database/index.ts` 幂等建表后追加幂等 ALTER（读 `PRAGMA table_info` 判存在 / catch 重复列）；`resources/schema.sql` 同步加列。旧行自动回填 `'ewei'`，向后兼容。
  - `account_version/shop_version` 是 ewei 专属；点大不需要。建议新增通用 `extra_json TEXT` 列存各商城专属参数（点大存 cookie/remember 等），逐步弃用 version 列（**不删旧列**）。
- `ewei-connectors.ts`：`EweiConnector/Summary`、`CreateConnectorInput`、`createConnector/updateConnector`、`resolveCredentials` 全部带上 `platform`（+`extra_json`）。
- 凭据/cookie 仍走 device-id 派生密钥本地加密、**排除云同步**（`sync/registry.ts`）。

### 4.2 协议层抽象为 MallAdapter + 注册表

- 新建 `services/mall/types.ts`：`MallAdapter` 接口（方法签名以现有 `ewei-client` 导出为准）：
  `login / listShops / switchShop / listGoods / getGoodsDetail / uploadImage / replaceGoodsImage / listGoodsCategories / addGoods`
  + **能力位** `capabilities: { needsShopSwitch, needsCaptcha, supportsAddGoods, detailFormat: 'html'|'blocks' }`（UI 据此显隐"选门店"、是否弹验证码等）。
- 现 `ewei-client.ts` 原样收进 `services/mall/ewei-adapter.ts`（**ewei 逻辑零改动**，`buildEditPayload` 多规格防误删整体平移）。能力位：`needsShopSwitch:true, needsCaptcha:false, detailFormat:'html'`。
- 新建 `services/mall/dianda-adapter.ts`：按 §1 协议实现。能力位：`needsShopSwitch:false, needsCaptcha:true, detailFormat:'blocks'`。`listShops/switchShop` 空实现。
- 会话层（`sessionStore`/`getState`/`ensureSession`）上移为商城无关的通用层；ewei 注入 AES/session-id 登录，点大注入 cookie-jar 登录。
- 新建 `services/mall/registry.ts`：`getAdapter(platform): MallAdapter`。

### 4.3 IPC 按 platform 分发（保持信道兼容）

- `ipc/index.ts` 每个 `ewei:*` handler 先取 `connector.platform` → `getAdapter(platform).xxx()`。**`ewei:*` 信道名与 `api.ewei` 保留不动**（对 renderer 透明，降低跨端改名风险）。`createConnector` 透传 `platform`。

### 4.4 验证码交互（回答"能否让用户自己填验证码"）

**能，且这是点大登录的必需设计。** 把登录泛化为可中断的两步：

1. `beginLogin(connectorId)`：适配器建立会话。ewei 直接静默登录返回成功；点大 GET 登录页拿 PHPSESSID + GET `/?s=/captcha` → 返回 `{ needCaptcha:true, captchaImage:<base64 png> }`。
2. renderer 弹**验证码对话框**（显示图 + 输入框 + "看不清换一张"=重新拉 `/?s=/captcha`）。
3. `submitLogin(connectorId, {captcha})`：点大 POST `username/password/captcha/remember=1`；`status==2`(验证码错)→ 回 `{needCaptcha:true, captchaImage}` 让用户重输；成功 → 持久化 cookie（加密、排除同步）。
- **降低验证码频次**：`remember=1` + 持久化 PHPSESSID/remember cookie，会话跨重启存活；仅会话真失效时才再弹验证码。
- ewei 的 `EweiError.showCaptcha` 字段已是同类交互雏形，复用该模式。
- 会话中途失效：ewei 静默重登；**点大无法静默重登** → 捕获到登录态失效时，向 renderer 抛"需重新登录（输验证码）"，而非静默失败。

### 4.5 渲染层 UI（单一入口 + 平台维度）

- **保持「店铺商品图」单一侧栏入口**（不开两个入口）。`MainLayout.vue` 门控改 `requireAnyPermission:['allow_ewei_shop','allow_dianda_shop']`（`passesPermissionFilter` 已支持数组 OR）。
- 连接器创建表单加**平台选择(ewei/点大)**（编辑时平台只读，平台决定协议不可中途改）；未被授权的平台选项禁用。连接器卡片显示平台标签。
- 商城名按连接器 `platform` 解析：`connectorMallName(c) = permissions.shops[c.platform]?.mall_name || '商城'`（兼容旧平铺字段）。
- `EweiConnectorsView` 的 `allowed` 改 `allow_ewei_shop || allow_dianda_shop`。"选门店"按钮按适配器 `capabilities.needsShopSwitch` 显隐（点大隐藏）。
- `stores/cloud-auth.ts`：`CloudPermissions` 增 `shops` 结构（+ 兼容保留 `allow_ewei_shop`/`ewei_shop_mall_name`）；初始默认 / logout 重置两处同步补字段。

## 6. 迁移铁律 / 兼容 / 部署顺序

1. **Migration 铁律**（两端）：已发布 migration 不改/删/重命名；只新增；Laravel 端只用 `Schema::`/`DB::` 不 import Model；桌面端 sqlite 用幂等 ALTER + 同步 `schema.sql`。
2. **向后兼容**：auth-check 保留 `can_use_ewei_shop`；下发保留平铺 `allow_ewei_shop`/`ewei_shop_mall_name`；连接器旧行回填 `platform='ewei'`。存量已授权 ewei 的客户端在 migration 内回填进关联表，避免掉权。
3. **部署顺序**：**先 agent-build 后端 → 再 agent-admin 后端 → 最后桌面端**。否则点大授权位读不到，点大 fail-closed（永远不可用）。
4. **点大特有**：验证码（必须用户参与）、可能的域名授权校验/频控；`buildEditPayload` 类防误删逻辑点大要重新踩点。
