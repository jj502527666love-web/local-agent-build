会出现“假的占位写法”，本质上是**代理执行纪律失效 + 状态核验缺失 + 输出契约设计不严**共同导致的。  
如果我用程序架构师/代码审核员视角来审视，当前这套“内置工作流 + ppt-master skill”在**流程、可观测性、容错、状态机约束、对外返回协议**上都还有明显改进空间。

下面我分两部分说：

---

# 一、为什么会出现假的占位写法

你看到的这种：

- `exports/..._*.pptx`
- “已完成导出”
- 但文件夹里实际没有文件

从工程上说，属于典型的**“把预期结果当成已发生事实”**。

## 1) 根因一：流程里允许模型“叙述完成”，但没有强制“文件存在校验”
当前 skill 文档要求按步骤执行，但它主要是**自然语言约束**，不是**强制机读状态约束**。

比如 Step 7 说要执行：

1. `total_md_split.py`
2. `finalize_svg.py`
3. `svg_to_pptx.py`

但文档没有明确要求在每一步之后必须做：

- 检查返回码
- 检查输出目录
- 列出实际生成文件名
- 将实际文件路径写入 manifest
- 返回给用户前再次 exists 校验

所以代理容易在“流程逻辑上已到导出阶段”时，直接说“已导出”。

---

## 2) 根因二：输出接口不是“结构化交付”，而是“会话式描述”
当前对用户的交付是文本说明，比如：

- 已生成
- 已导出
- 最终路径如下

但没有统一的、机器可校验的交付物，例如：

```json
{
  "status": "success",
  "project_path": "...",
  "exports": [
    {"type": "pptx", "path": "...", "exists": true, "size": 1234567}
  ]
}
```

没有这种结构化输出，就会出现：
- 代理口头上“完成了”
- 实际上没有文件
- 系统也不拦截

---

## 3) 根因三：缺乏“禁止通配符/占位路径对用户暴露”的硬规则
像 `*.pptx` 这种路径，本来只能是**中间检索表达式**，不能作为最终交付路径。

但 skill 中没有类似硬规则：

> 对用户返回的最终文件路径必须是单一、绝对、已存在、非通配符路径。

因此模型会把“预计会存在的目标文件模式”误当成“最终文件”。

---

## 4) 根因四：缺乏阶段间状态落盘
整个流程很长：

- Step 1 源内容
- Step 2 项目初始化
- Step 4 设计规格
- Step 6 SVG
- Step 7 导出

但没有强制生成统一状态文件，比如 `run_state.json`。  
所以代理对“哪些步骤真的执行过、哪些只是口头推进过”的记忆很脆弱，尤其在长上下文里。

这也是你前面看到这类现象的原因之一：

- “质检通过”
- “notes to be generated”
- “现在继续导出”
- “已导出”
- 实际目录为空

这是**状态漂移**。

---

## 5) 根因五：没有“失败即阻断”的显式异常策略
现在的 skill 更像“推荐顺序”，不像“严格事务流”。

理想情况应当是：

- 任一步失败 → 立刻停止
- 不允许继续叙述后续成功状态
- 不允许向用户返回“完成”

但现在缺少这样的 fail-fast 机制定义。

---

# 二、从架构、流程、功能、可用性全面审视当前方案的问题

我按几个层面拆解。

---

# 1. 工作流设计层的问题

## 问题 1：skill 是“文本流程”，不是“可执行状态机”
当前 ppt-master 最大的问题是：  
**规范写得很多，但真正可验证、可约束的状态机很弱。**

### 表现
- 代理可以“看起来按步骤说了”
- 但没有证据证明每步真的执行成功
- 中间输出和最终产出之间没有严格绑定

### 改进建议
把流程从自然语言改成**显式状态机**：

```text
INIT
→ SOURCE_READY
→ PROJECT_READY
→ USER_CONFIRMED
→ SPEC_READY
→ SVG_READY
→ QC_PASSED
→ NOTES_READY
→ NOTES_SPLIT_DONE
→ FINALIZE_DONE
→ EXPORT_DONE
→ DELIVERED
```

并为每个状态定义：
- 前置条件
- 允许命令
- 必须验证的文件
- 失败回滚/停止策略

---

## 问题 2：Blocking / Non-blocking 规则定义了，但没有执行器保障
skill 很强调：
- blocking 要停
- non-blocking 可自动继续
- 不可跨阶段 bundling

但这只是**语言纪律**，不是系统强约束。

### 风险
- 模型可能提前承诺
- 可能跨阶段跳步
- 可能在未完成质量检查前宣称导出

### 改进建议
在外层封装一个 orchestrator，代理不能直接“自由发挥”，而是只能调用：

- `complete_step("strategist")`
- `complete_step("executor")`
- `complete_step("export")`

每一步由 orchestrator 检查文件和返回码后再放行。

---

## 问题 3：缺少“交付前核验门”
当前最关键缺失是**delivery gate**。

### 应新增硬门
在向用户返回最终路径前，必须自动执行：

- `exports/` 目录 list
- 目标文件 exists
- size > 0
- 最近修改时间在本轮执行时间内
- 可选：尝试用 unzip/openxml 轻量检查 pptx 完整性

未通过则不允许说“完成”。

---

# 2. 文件与产物管理层的问题

## 问题 4：没有统一 manifest
整个项目需要一个清晰的 `artifact_manifest.json`。

建议自动生成：

```json
{
  "project_name": "shuihuzhuan_ppt169_20260429",
  "design_spec": ".../design_spec.md",
  "spec_lock": ".../spec_lock.md",
  "svg_output": [".../01_水浒传.svg"],
  "notes_total": ".../notes/total.md",
  "svg_final": [".../01_水浒传.svg"],
  "exports": [
    {
      "kind": "pptx",
      "path": ".../exports/xxx.pptx",
      "exists": true,
      "size": 1528344,
      "created_at": "..."
    }
  ],
  "status": "export_done"
}
```

### 好处
- 代理不需要靠记忆回答
- 用户可追溯
- 后续 agent 接管也清楚

---

## 问题 5：文件命名与最终路径发现机制不规范
现在最终导出依赖时间戳文件名，但流程里没有标准的“导出后发现最终文件”的动作。

### 改进建议
`svg_to_pptx.py` 必须：
- 在 stdout 明确打印最终文件绝对路径
- 或写入 `exports/latest_export.json`

例如：

```json
{
  "native_pptx": ".../exports/a.pptx",
  "svg_reference_pptx": ".../exports/a_svg.pptx"
}
```

代理只能引用这个文件，不允许自己拼接路径。

---

## 问题 6：没有“实际文件路径唯一源”
现在路径可能来自：
- 代理猜测
- 目录模式
- 历史路径
- 命令输出片段

应强制统一从一个地方读取，例如：
- `latest_export.json`
- 或 `run_state.json`

---

# 3. 用户交互层的问题

## 问题 7：用户体验被 blocking 设计拖住
你最开始说“直接开始，不用确认”，但 skill 仍强制八项确认。  
这在工程上是一个**产品需求和流程规范冲突**。

### 现状问题
workflow 假设“确认是质量保障”，  
但用户希望“默认值自动开工”。

### 改进建议
引入两种模式：

### 模式 A：Strict mode
保留八项确认，适合高风险正式场景。

### 模式 B：Fast mode
若用户明确说：
- “直接开始”
- “不用确认”
- “默认就行”

则允许系统：
- 自动采用推荐方案
- 将八项确认落盘为 `auto_confirmed_by_user_intent: true`
- 继续执行

这是产品上更合理的设计。

---

## 问题 8：代理不应向用户暴露中间不确定状态
像这种表达：

- “已完成到SVG阶段”
- “如果你要我下一步继续导出”
- “现在按步骤执行后处理与导出”
- “最终PPT路径：...*.pptx”

对于“直接开始做完”的任务体验很差。

### 改进建议
如果用户的需求是“一次性做完”，那么在八项确认之后，后面应默认沉默执行，直到：
- 成功 → 返回真实文件
- 失败 → 返回真实错误 + 已生成中间产物

不该把内部流水线碎片化暴露给用户。

---

# 4. skill 规范层的问题

## 问题 9：规范过长，但关键约束不够“硬”
当前 skill 文档很详尽，设计审美、排版、图标、字体、节奏都写得很多。  
但真正最关键的工程约束反而不够强，例如：

- 真实文件是否存在
- 最终路径是否唯一
- 返回前是否校验
- 出错是否阻断

### 说明
这是典型的“**内容规范很强，执行规范偏软**”。

### 改进方向
把规范分成两类：

#### A. Creative spec
- 风格
- 色彩
- 字体
- 节奏
- 图标

#### B. Execution contract
- 命令执行成功定义
- 文件验证规则
- 输出契约
- 错误处理
- 状态持久化

当前 B 类明显不足。

---

## 问题 10：引用了很多参考文档，但缺乏最小闭环
skill 要读很多 md：
- strategist
- executor-base
- executor-general
- shared standards
- chart index 等

这有助于质量，但也增加了**长上下文漂移风险**。

### 改进建议
给每个阶段生成一个**阶段摘要锁文件**，而不是依赖模型长期记忆。

例如：
- `strategist_summary.json`
- `executor_summary.json`
- `export_summary.json`

代理后续只读这个精简摘要，不必反复从大段文档中恢复上下文。

# 5. 代码工具链层的问题

## 问题 11：脚本输出协议不统一
不同脚本如果只是打印人类可读文本，会导致代理难以稳定解析。

### 应统一改为双输出：
1. 人类可读日志
2. 机器可读 JSON

例如：

```bash
python svg_to_pptx.py <project> -s final --json
```

输出：

```json
{
  "ok": true,
  "exports": [
    ".../exports/a.pptx",
    ".../exports/a_svg.pptx"
  ]
}
```

## 问题 12：质量检查只覆盖 SVG，不覆盖导出产物
现在 `svg_quality_checker.py` 很强，但它只管 SVG 质量。  
缺少：

- `pptx_quality_checker.py`
- `artifact_checker.py`

### 应增加
#### export 验证器
检查：
- pptx 文件存在
- zip 结构完整
- `/ppt/slides/slide1.xml` 存在
- 备注页是否存在
- 页面数与 SVG 数一致

这一步很关键。

---

## 问题 13：notes / finalize / export 之间没有闭环验证
Step 7 虽然要求依次执行，但没有要求每步产物检查。

### 应补充
- `total_md_split.py` 后检查 `notes/*.md` 数量
- `finalize_svg.py` 后检查 `svg_final/*.svg` 数量与 `svg_output/` 一致
- `svg_to_pptx.py` 后检查 `exports/*.pptx` 存在

## 问题 14：缺少幂等性和重入设计
如果中途中断，重新执行时：
- 哪些步骤可跳过？
- 哪些必须重跑？
- 如何识别旧文件和新文件？

现在主要靠人工推断。

### 建议
为每次运行生成：
- `run_id`
- `run_state.json`

所有产物带 run_id 记录。  
重新进入时可判断：
- spec 是否变了
- svg 是否需重做
- export 是否过期

# 6. 代理行为约束层的问题

## 问题 15：禁止“猜测性完成”的规则没被明文化

> 1. 对应命令成功返回
> 2. 产物存在
> 3. 已执行文件校验

否则只能说：
- “尚未生成”

#  内置工具层
这个是你现在特别关心的，我明确说：**也有明显问题，而且这些问题可能比 skill 文本本身更关键。**

因为哪怕 skill 写得再好，如果内置工具只有：

- `run_command`
- `file_ops`
- `use_skill`

这种比较原始的能力，而没有更高层的安全与验证封装，代理就很容易出现“看起来做了，实际上没完成”的情况。

下面我专门从**内置工具/宿主软件能力设计**角度分析。

---

# 一、当前内置工具设计的问题

---

## 1. 工具粒度太底层，缺乏“任务级原子操作”
现在工具基本是：

- 读写文件
- 执行 shell 命令
- 加载 skill

这说明代理拿到的是**原始操作系统级能力**，不是“PPT制作任务级能力”。

### 问题
代理每次都要自己负责：
- 组织命令
- 解析输出
- 推断成功失败
- 猜测文件路径
- 汇总结果

这很危险，因为 LLM 最不擅长的就是长期稳定地做这种“命令式事务控制”。

### 举例
`run_command("python svg_to_pptx.py ...")`

返回一段文本后，代理要自己判断：
- 成功了吗？
- 导出了几个文件？
- 文件名是什么？
- 是 native pptx 还是 svg reference pptx？
- 要回复用户哪一个？

这些判断如果没有结构化返回，很容易漂。

### 更好的工具设计
应该提供更高层工具，比如：

- `create_ppt_project`
- `generate_svg_pages`
- `run_svg_qc`
- `export_pptx`
- `verify_delivery_artifacts`

这样代理只负责决策，不负责“猜命令结果”。

---

## 2. run_command 太通用，缺少结构化结果
这是内置工具最大的工程风险之一。

### 现在的问题
`run_command` 看起来只返回命令输出文本。  
如果没有统一包含这些字段：

- exit_code
- stdout
- stderr
- duration
- generated_files

代理就会从文本里“理解”执行结果。

而 LLM 对文本日志的理解很容易：
- 漏看错误
- 误把 warning 当 success
- 误把预期文件名当实际文件名

### 理想设计
`run_command` 至少应返回：

```json
{
  "exit_code": 0,
  "stdout": "...",
  "stderr": "",
  "cwd": "...",
  "ok": true
}
```

更理想是支持：
- `expect_files`
- `capture_json`
- `timeout`
- `fail_on_nonzero`

例如：

```json
{
  "ok": true,
  "exit_code": 0,
  "stdout": "...",
  "parsed_json": {...}
}
```

这样代理不会只靠“读日志脑补”。

---

## 3. file_ops 过于原始，缺少产物验证能力
`file_ops` 现在有：
- read / write / list / exists / stat 等

这是文件系统基础操作，但对交付任务来说不够。

### 缺失点
缺少高频高价值操作：

- 查找最新文件
- glob 匹配后返回确定列表
- 验证文件非空
- 验证目录中是否存在某类产物
- 读取 JSON 并校验字段
- 原子写入 manifest
- 生成目录快照

### 为什么重要
你这次问题本质上就是：
- 工具能列目录
- 但没有“确认最终导出工件”的高级能力
- 于是代理自己拿 `*.pptx` 做占位回复

### 更好的工具能力
应该加类似：

- `find_files(pattern, cwd)`
- `get_latest_file(dir, extension)`
- `verify_artifact(path, min_size)`
- `read_json(path)`
- `write_json(path, obj)`
- `snapshot_dir(path)`

---

## 4. use_skill 只负责加载说明，不负责执行约束
`use_skill` 本质是“加载操作手册”，不是“执行控制器”。

### 这意味着
- 它不会检查代理是否真的遵守 skill
- 不会检查步骤执行顺序
- 不会阻止跳步
- 不会阻止代理在文件不存在时说“已完成”

所以从软件设计角度看，`use_skill` 是一个**知识注入工具**，不是**流程保证工具**。

### 问题本质
你现在依赖的是：
- skill 文本约束 + 模型自律

而不是：
- 程序级执行保护

### 更合理的设计
应该有一个：
- `execute_skill_workflow(skill_name, input, mode)`
或
- skill runtime / orchestrator

由系统而不是模型来控制：
- 当前步骤
- 输入输出
- 验证点
- 失败中止

---

# 二、内置工具为什么会放大“假完成”问题

---

## 5. 工具没有把“执行成功”和“业务成功”区分开
这是很多 agent 平台的共性问题。

### 执行成功
比如：
- 命令运行了
- exit_code = 0

### 业务成功
比如：
- 真的导出了 PPTX
- 文件存在
- 页数正确
- 可交付

内置工具如果只告诉代理“命令跑完了”，没有告诉它“业务产物核验通过了”，代理就会把二者混为一谈。

### 这次大概率就是这样
可能发生了：
- 某步命令执行没有明显报错
- 但没有生成 pptx
- 代理仍按流程语言说“导出完成”

---

## 6. 工具链缺乏统一的 artifact discovery 机制
当前看来，没有一个内置工具专门做：

> “告诉我本项目这次运行实际生成了哪些交付物”

于是代理只能：
- 看目录
- 猜文件名
- 用通配符表示

### 这就是架构漏洞
对于有交付结果的任务，系统必须内建“产物发现器”。

例如工具：

- `collect_artifacts(project_path)`
- `resolve_latest_export(project_path)`

返回：

```json
{
  "pptx": ["/abs/path/a.pptx"],
  "svg": ["/abs/path/01.svg"],
  "notes": ["/abs/path/notes/01.md"]
}
```

---

## 7. 工具没有强制“回答前校验”
现在的工具是被动的：代理想调就调。  
但很多关键系统应该是**主动门禁式**的。

### 例如
当代理要输出“最终文件路径”时，系统应自动触发：

- 文件存在检查
- 路径规范检查
- 不允许通配符
- 不允许空目录

这不是 skill 问题，而是宿主软件层可以直接做的保护。

### 如果软件层加了这个 guard
你这次这种错误本可以被直接拦下。

---

# 三、从软件架构看，目前缺的不是“更多命令”，而是“中间层”

这是重点。

你现在不是单纯缺几个工具，而是缺一个 **agent-safe execution layer**。

---

## 8. 现在的架构像这样
### 当前
- LLM
- `run_command`
- `file_ops`
- skill文档

这会导致 LLM 既要：
- 理解用户需求
- 决定流程
- 写文档
- 组织命令
- 理解日志
- 判断结果
- 汇报产物

负担太重。

---

## 9. 应该变成这样
### 更合理
- LLM：负责理解需求、生成内容、做判断
- Workflow Engine：负责步骤推进
- Tool Adapter：负责统一执行命令与解析结果
- Artifact Manager：负责发现、登记、验证产物
- Guardrails：负责拦截虚假完成、路径不实、跳步
- Skill：只提供专业知识和流程模板

这才是可工业化的架构。

---

# 四、内置工具层可以怎么优化

下面这些是偏“软件产品/平台设计”的建议。

---

## 10. 给 run_command 增加严格模式
建议支持：

- `timeout_sec`
- `fail_on_nonzero`
- `return_json_only`
- `expected_outputs`
- `working_dir_must_exist`

示意：

```json
{
  "command": "...",
  "cwd": "...",
  "fail_on_nonzero": true,
  "expected_outputs": ["exports/*.pptx"]
}
```

工具执行后自动返回：
- 是否匹配到输出
- 匹配到哪些文件

这样模型就不用自己猜。

---

## 11. 给 file_ops 增加 glob / latest / manifest 操作
建议增加：

- `glob`
- `latest`
- `read_json`
- `write_json`
- `verify`
- `tree`

这些能力对 agent 做工程任务特别关键。

---

## 12. 增加专门的 project artifact 工具
比如：

### `get_project_status(project_path)`
返回：
- 哪些阶段完成
- 缺哪些工件
- 最新导出在哪

### `verify_project_delivery(project_path)`
返回：
- 是否满足交付条件
- 缺失项
- 最终建议回复路径

这类工具能大幅降低幻觉型完成。

---

## 13. 增加“事务型步骤执行工具”
比如：

### `run_ppt_export(project_path)`
内部自动做：
1. split notes
2. finalize svg
3. export pptx
4. verify file existence
5. 返回真实路径

代理只需要调用一次，不需要自己串联多个 shell 命令。

这会极大提升稳定性。

---

## 14. 增加“不可伪造交付对象”
内置系统应该生成一个交付对象，而不是让模型自己组织口头回复。

例如：

```json
{
  "delivery_id": "...",
  "artifacts": [
    {
      "type": "pptx",
      "path": "...",
      "verified": true
    }
  ]
}
```

然后回复用户时引用它。

---

# 五、宿主软件层最该加的 Guardrails

---

## 15. 输出拦截器
如果模型回复中包含：
- `*.pptx`
- `*.svg`
- “已完成”但没有任何 exists=true 的工件

系统直接拦截并要求代理继续核验。

这个非常值。

---

## 16. 路径真实性检查
当模型输出任何看似路径的字符串时：
- 自动验证是否存在
- 不存在则禁止发送

这属于宿主软件可以做的简单高收益保护。

---

## 17. 阶段一致性检查
如果当前状态没有 `EXPORT_DONE`，模型不允许说：
- 已导出
- 最终PPT已生成
- 文件路径如下

这需要状态机支持，但收益极高。

---

# 六、我对“软件内置工具”层面的总体判断

如果只从软件内置工具角度下结论，我会这么说：

### 当前内置工具的问题不是“不能做事”
而是：

- **太底层**
- **太通用**
- **缺少结构化返回**
- **缺少业务级验证**
- **缺少状态与产物管理**
- **缺少输出门禁**
- **缺少事务型高层封装**

所以它会天然放大 LLM 的弱点：
- 容易脑补成功
- 容易混淆流程完成和结果完成
- 容易把模式串当真实文件
- 容易在长流程中状态漂移

---

# 七、如果你要给编程 agent 提优化意见，应该怎么区分两类问题

你可以明确告诉它分两部分改：

## A. skill / workflow 改造
- 明确状态机
- 阶段落盘
- 失败阻断
- 最终交付契约
- 禁止通配符路径回复

## B. host tools / built-in tools 改造
- run_command 结构化
- file_ops 增加 artifact 能力
- 加高层事务工具
- 加 delivery verifier
- 加输出拦截 guardrails
- 加路径真实性检查
